# 지출 라이프사이클

> 유형: 정책 | 관련 entity: `exp-cst-process`, `exp-oact-process`, `exp-res-process`, `exp-pym-cmd-process`

## 개요

한 건의 지출은 등록부터 자금 집행까지 4단계를 거칩니다. 각 단계는 별도의 화면, 별도의 결재자, 별도의 상태코드를 가지지만 PK 체인(EXP_CST_NO → EXP_OACT_NO → EXP_RES_NO → EXP_PYM_CMD_NO)으로 연결됩니다.

## 4단계

```
┌───────────┐    ┌────────────┐    ┌───────────┐    ┌───────────────┐
│  품의     │ →  │ 원인행위   │ →  │  결의     │ →  │  지급명령      │
│ (EXP_CST) │    │ (EXP_OACT) │    │ (EXP_RES) │    │ (EXP_PYM_CMD) │
└───────────┘    └────────────┘    └───────────┘    └───────────────┘
   CST_STAT       OACT_STAT          RES_STAT           CMD_STAT
   A64 코드       A68 코드           A69 코드           A61 코드
```

### 1. 품의 (CST)

부서가 "이만큼 쓰겠다"고 신청하는 단계. AccExp102BMulti 화면에서 다음을 입력합니다.

- 사용 부서 / 회계연도 / 사업 / 예산과목 × N
- 거래처 × N
- 과목별 분개유형(JRNLZ_TY_CD) × N

저장 시 즉시:
- BDG_EBDG_CPL의 잔액 검증·차감 (cross-domain)
- EXP_CST + EXP_CST_SBJ + EXP_CST_CLT INSERT
- ACC_JMT로 분개유형 매핑을 조회해 ACC_SLIP + ACC_SLIPDTS 부채전표 생성 (`SLIP_STAT='M'`)
- "결재상신" 클릭 시 CST_STAT을 `01`로 잡고, SP가 즉시 `91`로 덮어쓴다

### 2. 원인행위 (OACT)

품의 결재 완료(CST_STAT=93)된 건에 대해 외부 계약·발주 등의 법적 근거를 기록.
다중과목 화면(102BMulti)에서는 같은 트랜잭션에 OACT를 동시 생성하므로 단계 분리가 약합니다.
OACT_STAT(A68)는 CST_STAT과 쌍을 이루어 전이합니다.

### 3. 결의 (RES)

회계팀이 지출을 회계상 확정짓는 단계. AccExp120Multi에서 처리합니다.

- RES_STAT: 02(등록) → 88(검토완료) → 93(승인) / 15(반려)
- 승인 시 ACC_SLIP.SLIP_STAT을 `'M'`(미승인) → `'S'`(승인)로 전이
- APP_STAT은 `01`(미결재) → `93`(결재완료)

### 4. 지급명령 (PYM_CMD)

자금팀이 실제 자금 집행을 지시하는 단계. AccExp106Multi에서 처리.

- CMD_STAT: 01(미결재) → 91(상신) → 93(전결) / 15(반려)
- 결재완료/전결 시 SP 호출로 **52번 명령전표** 생성
- PYM_STAT을 `M`(미지급) → `J`(지급완료)로 전이

## 단계 간 게이트

다음 단계는 이전 단계가 결재 완료된 후에만 진입할 수 있습니다.

| 진입 단계 | 직전 단계 조건 |
|----------|---------------|
| 원인행위 | CST_STAT = 93 |
| 결의 | OACT_STAT = 93 |
| 지급명령 | RES_STAT = 93, APP_STAT = 93 |

## 흔한 실수

- 결재 SP(PKG_EA_RECOM)가 변경한 상태값(91/92/93)을 모르고 서버 검증에서 `01`만 허용 → 상신 직후 SP가 91로 바꾼 뒤 모든 후속 액션이 차단
- 다음 단계 시작 후 직전 단계를 수정하려고 시도 → 상태 머신이 막아야 정상
- 다중과목 화면에서 일반 화면 흐름을 가정 (실제는 한 트랜잭션에 OACT 동시 생성)
