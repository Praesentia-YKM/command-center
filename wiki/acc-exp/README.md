# 회계/지출 (acc-exp)

> 유형: 도메인 | 관련 레포: ALLSP_STD | 패키지: `src/java/com/allsharp/acc/exp`

## 배경

회계 ERP에서 가장 호출 빈도가 높고 결재·감사·전표가 모두 얽히는 도메인입니다.
한 건의 지출이 발생하면 **품의 → 원인행위 → 결의 → 지급명령** 4단계를 거쳐 자금이
실제로 빠져나가며, 매 단계마다 결재 워크플로우와 회계전표(ACC_SLIP) 갱신이 동반됩니다.
다중 예산과목 + 다중 거래처를 한 트랜잭션에서 처리해야 하는 **Multi 화면**(102B/120/106)이
실무 운영의 핵심입니다.

## 풀려는 문제

- **예산 통제와 정합성**: 지출이 발생하는 즉시 BDG_EBDG_CPL 잔액을 정확히 차감해야 합니다.
  공급가액 기준 통제와 총금액 기준 전표가 한 화면에서 공존하므로 둘을 혼동하면 안 됩니다.
- **결재 상태 머신**: CST_STAT(공통코드 A64), OACT_STAT(A68), RES_STAT(A69), CMD_STAT(A61)이
  각 단계의 상태를 표현합니다. 상신/전결/반려/취소가 잘못 매핑되면 전 단계가 막힙니다.
- **다중 분개유형**: 같은 예산과목이라도 분개유형(JRNLZ_TY_CD)에 따라 차변·대변이 다른 계정에
  잡힙니다. ACC_JMT 매핑이 화면 입력과 정확히 맞아야 전표가 양차로 떨어집니다.
- **DB 방언 3종**: 같은 SQL을 MariaDB / Oracle / MSSQL 매퍼에 동기화해야 합니다.
  NVL / IFNULL / ISNULL, ROWNUM / OFFSET, FN_* 함수 호출 규칙이 모두 다릅니다.
- **전표 FK 정리 순서**: ACC_SLIP을 참조하는 EXP_RES.PYM_SLIP_NO / RES_SLIP_NO를 먼저
  NULL로 해제하지 않으면 전표 삭제가 FK 위반으로 실패합니다.

## 사용자

| 사용자 | 어떻게 사용하는가 |
|--------|--------------------|
| **요청부서 담당자** | AccExp102BMulti(품의) 등록·상신, 반려된 건 수정 재상신 |
| **결재자** | 전자결재(esaApp) 화면에서 결재·전결·반려 처리 |
| **회계팀** | AccExp120Multi(결의) 검토·승인, ACC_SLIP / ACC_SLIPDTS 확정 |
| **자금팀** | AccExp106Multi(지급명령) 처리, 52번 명령전표 생성 |
| **감사·정산** | ACC_SLIP과 EXP_* 추적 조회 |

## 성공 기준

- 다중과목·다중거래처 한 화면 트랜잭션에서 5단계 금액 검증 모두 통과
- 결재 단계 SP(PKG_EA_RECOM$EA_RECOM_GUBUN)가 변경한 상태값(91/92/93)과 서버 검증 일치
- 회계전표 SLIP_AMT = 총금액, 예산 차감 = 공급가액 (혼동 0건)
- 3개 DB 방언 매퍼 결과 동일성 보장
